## 原则陈述

面对任何学习、技术选型、产品设计或职业决策时，**优先从现实问题出发寻找本质矛盾**，再选择最合适的工具解决问题；**不要**因为掌握了某个工具而寻找应用场景。

```text
Problem Statement → 价值/矛盾 → 方案 → 工具
```

**反序**（禁止默认）：

```text
工具 → 找场景 → 伪需求
```

---

## 行为表现

| 场景 | 先做 | 后做 |
|---|---|---|
| 学习 | 我要解决什么问题？ | 选课程 / 框架 / 教程 |
| 技术选型 | 约束与验收标准 | 对比 LangGraph / MCP / … |
| 产品设计 | 用户价值与核心矛盾 | 架构与实体设计 |
| 职业项目 | 作品集要证明什么能力 | 堆技术栈 |

**可观测动作**：

- 先写 **Problem Statement**，再写技术方案
- 先定义 **用户价值**，再讨论架构
- 先 **验证需求**，再优化工程
- 技术选型服务于问题，而不是相反

---

## 反模式

| 反模式 | 典型表现 |
|---|---|
| 为框架而项目 | 「学完 LangGraph 做个 demo」 |
| 为 Agent 而 Agent | 先定 Multi-Agent 再想找业务 |
| 为架构而架构 | 微服务 / 事件驱动无需求支撑 |
| 为 AI 而 AI | 凡事先上 LLM，不问是否必要 |

> 与 [[常犯的错误#探索型无边界|探索无边界]]、[[计划执行防护规则#五、探索型工作预算|探索型预算]] 同源：无 Problem 的探索易挤占主块。

## 决策自检（30 秒）

开口或开番茄前自问：

1. **Problem Statement 一句话写得出吗？**
2. 当前动作是在 **逼近问题**，还是在 **熟悉工具**？
3. 若去掉这个框架/技术，问题仍成立吗？

任一为否 → 停；回收件箱写清问题，再 4D。
