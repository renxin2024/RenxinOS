# RRF 算法讲解（Reciprocal Rank Fusion）

> 来源：RenxinOS V2 开发过程中的学习讨论
> 适用场景：混合检索（Hybrid Retrieval）结果融合

---

## 1. RRF 解决什么问题

在混合检索系统中，我们同时使用两路检索：

- **关键词检索**：返回整数分数（词重叠次数），如 `score=7`
- **向量检索**：返回浮点分数（余弦相似度），如 `score=0.54`

这两个分数**量纲不同，无法直接比较或相加**。  
RRF 的解法是：**不看分数，只看排名**。

---

## 2. 核心公式

$$
\text{RRF\_score}(doc) = \sum_{i} \frac{1}{k + \text{rank}_i}
$$

- `rank_i`：该文档在第 i 路检索结果中的排名（从 1 开始，第 1 名最相关）
- `k`：平滑常数（RenxinOS 设为 20），防止第 1 名分数过于"霸主"
- 所有路中未出现的文档，贡献值为 0（不参与求和）

---

## 3. 用真实数据走一遍

**查询**："手头事情太多怎么办"

| 路 | rank | 文档 | 原始分数 |
|---|---|---|---|
| 关键词 | 1 | ABC+4D工作法.md | 3 |
| 关键词 | 2 | GTD任务执行规范.md | 2 |
| 向量 | 1 | 时间任务管理统一工作流.md | 0.54 |
| 向量 | 3 | ABC+4D工作法.md | 0.49 |

**计算 `ABC+4D工作法.md` 的 RRF 分数（k=20）：**

```
关键词路 rank=1 → 1/(20+1) = 0.0476
向量路   rank=3 → 1/(20+3) = 0.0435
RRF 总分        = 0.0911
```

这与实际 trace 记录的 `top_rrf_score=0.0911` 完全一致。

---

## 4. 参数 k 的作用

| k 值 | 效果 |
|---|---|
| k 小（如 5） | rank=1 和 rank=2 的分差大，排名影响更明显 |
| k 大（如 60） | 各排名分差被拉平，结果更"保守" |
| k=20（RenxinOS） | 适合小型个人知识库的经验值 |

---

## 5. 为什么不用"排名相加取最小"（Borda Count）

"排名相加取最小"是另一种融合方案，叫 **Borda Count（博达计数法）**，逻辑上也合理，但有一个致命缺陷：

### 问题：如何处理"缺席"文档？

关键词路返回 5 条，向量路返回 8 条，大多数文档只出现在一路。

对于只在向量路出现的文档，它的关键词路排名应该是多少？

- 赋 rank=6（末位+1）？
- 赋 rank=100（全库末位）？
- 赋 rank=∞（无穷大）？

**三种选择结果完全不同，且都没有理论依据。**

### 对比示例

有 3 个文档，两路结果如下：

```
         关键词路    向量路
文档A     rank=1    rank=1    ← 两路都第一
文档B     rank=2    缺席      ← 只在关键词路
文档C     缺席      rank=2    ← 只在向量路
```

| 方案 | 文档A | 文档B | 文档C |
|---|---|---|---|
| Borda Count（缺席=∞） | 1+1=2 ✅ | 2+∞=∞ ❌无法排名 | ∞+2=∞ ❌无法排名 |
| RRF（k=20） | 1/21+1/21=0.0952 | 1/22+0=0.0455 | 0+1/22=0.0455 |

RRF 所有文档都有明确的可比分数。

### 根本原因

| 场景 | 适用算法 | 原因 |
|---|---|---|
| 每个候选人都有完整排名（如选举） | Borda Count | 所有人都投了票，没有缺席 |
| 各路检索只返回部分结果（如 RAG） | **RRF** | 文档可以不在某一路出现，缺席贡献 0 |

> **一句话**：Borda Count 需要假设"不在结果里等于多少名"，RRF 绕过了这个问题——缺席就是 0 贡献，数学上干净自洽。

## 6. RRF 在 RenxinOS 代码中的位置

```
retrieve_hybrid()
    ├── retrieve()           → kw_hits（按 keyword score 降序）
    ├── retrieve_semantic()  → em_hits（按 embedding score 降序）
    └── _rrf_merge()         → 两路合并，按 rrf_score 重新排序返回
```

对应 trace 页面的三个步骤：

```
keyword_retrieve → embedding_retrieve → rrf_merge → hybrid_decision(rrf_merged)
    29ms               1257ms              0ms            0ms
```

其中 `rrf_merge` 耗时接近 0ms，因为这是纯内存的数学计算，没有网络请求。

---

## 7. 延伸思考

- 如果只有一路有结果（关键词为空），RRF 退化为直接返回向量路结果
- 如果两路都很弱（keyword top_score=1），RRF 无法"无中生有"提升质量，根本问题在知识库覆盖度
- k 的最优值可以通过评测集调参，就像我们调 `KW_MIN_SCORE` 一样
