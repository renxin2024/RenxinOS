# 为什么指令遵循是核心指标？为什么有专门为 Agent 打造的模型？

> 问题来源：从上一条"为什么模型能调用工具"的追问
> 关联：`docs/learning/v3-why-llm-can-call-tools.md`、`src/agent_raw/tools.py:218`

## 1. 核心答案（一句话）

Agent 场景下，模型的输出**不再面向用户，而是面向另一段代码**。"指令遵循"衡量的是模型能否稳定输出调用方代码能解析的格式；"Agent 专用模型"就是针对这个需求 fine-tune 出来的。

## 2. Agent 场景的本质 —— 输出是"指令"，不是"回答"

回顾上一条笔记的核心认知：

```
模型 → 输出文本 → 调用方解析 → 执行工具 → 结果注入上下文
```

这意味着模型的输出要经过**机器解析**。机器解析和人类阅读的要求完全不同：

| 人类阅读 | 机器解析 |
|---|---|
| "大概这个意思就行" | 字段名、类型、结构必须完全一致 |
| 多一点少一点文字无所谓 | 多一个逗号 JSON 就炸了 |
| 语气、措辞变化增强阅读体验 | 每次格式变化都是错误 |

类比 Java：你给前端返回 JSON，如果有时 `{userName: "..."}` 有时 `{user_name: "..."}` 有时直接返回 "用户名是张三"——前端反序列化全炸。Agent 场景同理。

## 3. 指令遵循测的到底是什么

指令遵循考察的是模型在以下场景的表现：

### 3.1 格式约束
- 要求输出 JSON → 输出合法 JSON、字段名对、类型对
- 要求输出列表 → 输出列表而不是散文
- 要求用特定分隔符 → 用对分隔符

### 3.2 行为约束
- 要求只回答笔记内容 → 不编造笔记中没有的东西
- 要求不知道就说不知道 → 不强行编答案
- 要求使用工具 → 不走捷径直接回答

### 3.3 多轮约束
- 要求记住上一步的结果 → 不丢失上下文
- 要求不要重复调用同一个工具 → 不无限循环
- 要求达到条件时终止 → 不提前也不延迟

## 4. 指令遵循不好在 Agent 场景的后果

对照你 V3 代码的每个环节：

| 指令遵循问题 | V3 代码受到的影响 | 对应位置 |
|---|---|---|
| 模型跳过工具直接回答 | `parse_tool_call()` 返回 None，进入终止条件 2，把"编造的回答"当最终答案 | `react_loop.py:243` |
| 模型输出不合法 JSON | JSON 解析失败，回退到文本格式但照样解析不了 | `tools.py:253` |
| 模型编造 Observation | `stop=["Observation:"]` 就是为了防这个 | `react_loop.py:211` |
| 模型重复调用同一工具 | `max_steps` 超限，终止条件 3 触发 | `react_loop.py:285` |
| 模型在不需要时硬塞 tool_call | 执行到不存在的工具，报错后传给下一轮 | `tools.py:168` |

## 5. 为什么 Agent 专用模型不是噱头

普通对话模型的训练数据是"人类对话"——开放、自然、多样。Agent 专用模型的训练数据是"工具调用序列"——精确、结构化、可预测。

### 5.1 训练数据的差异

| 普通模型 | Agent 模型 |
|---|---|
| 训练数据：人类对话（帖子、文章、QA） | 训练数据：工具调用序列 + 格式纠正对 |
| 奖励函数：人类觉得自然、有帮助 | 奖励函数：格式正确、工具选对、不该用时不用 |
| 核心能力：知识广、表达好 | 核心能力：听指令、格式稳定、不编造 |

### 5.2 对应到你的项目

V3 用的是通用模型 `deepseek-v4-flash`，所以需要在 prompt 里做大量约束（stop 序列、详细 tool 描述、严格格式要求）。如果换成 Agent 专用模型：
- `stop=["Observation:"]` 可能不再需要（模型天然不会编造工具返回）
- prompt 可以更简洁（模型对 tool schema 的理解更准）
- 解析逻辑可以简化（格式更稳定）

这恰好是 V4 要体会的东西——V3 手写时感受到"模型不听话"的各种痛点，V4 用框架 + 更强的模型后，观察哪些问题自然消失了。

## 6. 面试讲法

**30 秒版**：指令遵循衡量的是模型能否稳定输出"给机器看"的格式。Agent 场景下模型输出要经过代码解析，格式不稳定直接导致执行失败。Agent 专用模型用工具调用序列做 fine-tune，解决了普通模型"会说话但不听话"的问题。

**2 分钟版**：
1. 底层逻辑承接上一条：模型输出在 Agent 场景不是给用户看的，是给 `parse_tool_call()` 解析的
2. 指令遵循差 = JSON 不合法 / 跳过工具 / 编造返回值 / 重复调用 → 对应 V3 代码 4 个终止条件
3. Agent 专用模型做的事：用工具调用序列 fine-tune，奖励格式正确而非内容好坏
4. 你的项目映射：V3 用通用模型所以需要大量 prompt 约束和 stop 序列；V4/V5 如果换更强的模型，约束可以松绑
5. 面试亮点：能讲清楚"通用模型 vs Agent 模型"的差异，而不是笼统说"这个模型更好"
