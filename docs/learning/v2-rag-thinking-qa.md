## RAG 评测思考题 Q&A

### Q1：为什么分数从 88.3% 降到 61.5% 反而是进步？

**A**：旧版 10 题全是「happy path」——关键词直接匹配、无负例、无口语化改写。新版 16 题加了负例题和同义改写，暴露了旧评测没发现的 3 个真实问题（系统不会拒答、口语化检索失效、Q9 检索偏差）。

> **核心认知**：评测分数下降 ≠ 系统退步。更严格的评测发现更多问题，这才是评测的价值——**发现盲区**，而不是刷高分。

---

### Q2：如果要加第 3 组同义改写，选哪个问题？

**A**：建议选 Q7（阅读笔记）或 Q6（分散类型）。

- Q7 原文：「读完一本书后怎么做笔记？分几步？」
- 改写 A：「看完一本书要整理哪些东西？」（口语化，无「笔记」术语）
- 改写 B：「怎么把书里的知识变成自己的？」（极口语化，完全换表述）

**原因**：Q7 涉及多阶段流程（首读→结构回顾→深度笔记），如果口语化版本搜不到，说明检索对流程性描述的泛化能力也需要提升。

---

### Q3：Embedding 接入后，哪类题提升最大？哪类可能不变？

**A**：

| 题型 | 预测变化 | 原因 |
|---|---|---|
| **paraphrase 同义改写** | **提升最大** | Embedding 能理解语义，「手头事太多」≈「4D 优先级排序」 |
| **fuzzy 模糊题** | **明显提升** | 口语化表述和笔记原文的语义距离更近 |
| **unanswerable 负例** | **可能不变或更差** | Embedding 可能把无关问题也匹配到高分（语义太近）→ 需要单独调阈值 |
| **single_hop 单跳** | **基本不变** | 关键词检索本来就很准，Embedding 是锦上添花 |

> **经验**：不是所有指标都会同时提升。评测的意义就是看**哪些升了、哪些降了**，而不是只看平均分。

---

### Q4：如果只改一个失败模式，先改哪个？

**A**：先改 **SEARCH_FAILURE**（占比 19%，3 道题）。

**原因**：
1. 它是最容易修的——Embedding 语义检索直接解决「关键词不匹配」的问题
2. SHOULD_REFUSE（拒答）需要加 score 阈值逻辑，依赖检索质量先稳定
3. CONTEXT_INCOMPLETE 需要改 chunk 策略，工作量更大

> **原则**：先修上游（检索），再修下游（生成）。先修高占比，再修低占比。

---

### Q5：「系统不会拒答」除了加 score 阈值，还有什么方案？

**A**：至少有 4 种方案，从简单到复杂：

| 方案 | 做法 | 优缺点 |
|---|---|---|
| **1. Score 阈值** | top-1 score < 5.0 时返回「未找到」 | 最简单，但阈值需调参 |
| **2. LLM 判断** | 把 top-k 结果和问题发给 LLM，让它判断「这些资料能回答这个问题吗？」 | 准确但多一次 API 调用 |
| **3. 置信度分类器** | 训练一个小模型判断 query-document 相关性 | 需要训练数据，但推理快 |
| **4. Prompt 引导** | 在 system prompt 中加「如果资料中没有答案，请明确告知用户」 | 零成本，但依赖 LLM 自觉遵守 |

> **推荐落地顺序**：先做方案 1（零成本快速验证）→ 如果不够再做方案 4（加 prompt 引导）→ 生产级再做方案 2 或 3。

---

### Q6：端到端评测分数低，怎么判断改检索还是改 prompt？

**A**：看 pipeline 各阶段的输出：

| 检索 Recall | 生成 Faithfulness | 诊断 | 改什么 |
|---|---|---|---|
| 高 | 高 | 端到端成功 | 不用改 |
| **低** | — | 搜错了文件 | 改检索（Embedding / 阈值 / chunk 策略） |
| 高 | **低** | 搜到了但 LLM 答错 | 改 prompt（指令更明确 / 换模型） |
| 低 | 低 | 两层都有问题 | **先修检索**（垃圾进垃圾出） |

> **核心原则**：永远先修上游。检索质量是生成质量的天花板——如果搜到的文件是错的，prompt 再好也救不回来。

---

### Q7：当前评测脚本没有调用大模型，为什么？

**A**：因为评测分两个阶段，当前只做了第一阶段。

```
用户提问 → [检索模块] → top-k 文档 → [LLM 生成] → 最终回答
              ↑                           ↑
          ✅ 已评                      ⏳ 待加
      只测「搜得准不准」              测「答得好不好」
```

**为什么分阶段**：先独立评检索，确保检索层没问题，再加生成评测。如果一上来就跑端到端，分数低了你分不清是「搜错了」还是「答错了」。

> **学习路径**：评测设计 → 检索基线 → 检索优化 → 生成评测 → 端到端评测。这就是 V2 S1 → S3 → S5 的演进顺序。

### Q8：评测集设计可能存在哪三类潜在问题？

**A**：按严重程度从外到内，有三层需要检查：

| 检查层 | 问题类型 | 表现 | 修复方法 |
|---|---|---|---|
| **检查1** | 期望文件不在知识库中 | `expected_source_files` 指向了知识库范围外的文件 | 确认文件在 `data/principle/` 下并已被 ingest |
| **检查2** | 关键词在笔记中找不到 | `expected_answer_keywords` 措辞与原文不一致（如缺顿号、多余字符） | 对照原文修正关键词，用子串匹配而非精确匹配 |
| **检查3** | ground_truth 缺失关键词 | 标准答案是重组的答案，遗漏了笔记中的部分关键信息 | 补充 ground_truth，确保所有关键词都被覆盖 |

> **核心认知**：评测数据本身有 bug，会让所有测试结论都失真。这叫 **ground truth pollution（标准答案污染）**。修复评测集的优先级不低于修复系统本身。

### Q9：题目的「问法」本身可能导致什么问题？

**A**：即使文件存在、关键词正确，问法不准也会导致「理论上无法召回」：

| 问法问题类型 | 举例 | 后果 |
|---|---|---|
| **术语错位** | 问「怎么拒绝任务」，但笔记用的是「Defer/Delete」 | 关键词检索全空，Embedding 相似度也低 |
| **跨文件假设** | 一道题期望同时召回 A 和 B，但 A 和 B 在语义上毫不相关 | 无论用哪种检索都只能召回其中一个 |
| **问题范围超出知识库** | 问了一个笔记里只有两句话、没有展开的话题 | 系统召回正确但上下文太少，LLM 会答得很浅 |
| **歧义问法** | 「怎么做好计划」——「计划」可以指 Day Planner、项目计划、决策文档 | 召回结果随机，无法稳定评测 |

> **设计原则**：写评测题前，先在原始笔记中找到「答案段落」，再根据段落内容反推题目的问法——从「答案 → 问题」，而不是「问题 → 找答案」。

---

### Q10：发现召回率不达预期时，正确的诊断流程是什么？

**A**：按以下顺序逐层排查，先验证数据层，再看算法层：

```
第1步：确认 expected_source_files 在知识库范围内（data/principle/）
       ↓ 是
第2步：grep 确认期望文件里真的有答案内容
       ↓ 是
第3步：检查是否是 chunk 划分导致答案被截断在两个 chunk 里
       ↓ 排除
第4步：才开始怀疑检索算法（阈值 / RRF 权重 / embedding 质量）
```

> **反面教训**：本次诊断 Q6 时，绕了一圈去排查 `gtd-harness-铁律.md` 是否缺失，后来才发现它根本不在 `expected_source_files` 里。**数据层没确认就跳到算法层调参，是最常见的浪费时间方式。**
