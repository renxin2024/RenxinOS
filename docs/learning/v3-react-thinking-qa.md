# V3 ReAct Loop 思考题与解析

> 基于 `src/agent_raw/react_loop.py` 的学习记录
> 创建日期：2026-06-22

## 思考题 1：如果 scratchpad 只保留最近 2 步，会发生什么？

**答案**：会损失上下文导致模型推理出错。

**解析**：
- LLM 没有会话机制，每次 API 调用都是独立的
- scratchpad 就是模型的"工作记忆"——它能看到的所有历史信息
- 如果只保留最近 2 步，模型会丢失之前的关键上下文

**可能出现的错误**：
- 重复执行已经做过的 Action（因为忘了做过）
- 忘记之前查到的关键 Observation
- 上下文不完整导致推理链条断裂

**类比**：就像考试时草稿纸只能写两行，擦掉前面的就忘了中间步骤。

## 思考题 2：如果 max_steps=5 后模型还没给 Final Answer，返回的 thought 里大概是什么？

**答案**：取决于模型为什么没给 Final Answer。

**情况 A：模型一直在思考但没行动**
- thought 里可能是「我需要查笔记……」「让我看看任务池……」
- 模型卡在"想"的阶段，没有执行工具获取信息

**情况 B：模型调了工具但工具返回空/错误**
- thought 里可能是「没找到相关信息，我需要换个关键词……」
- 工具调用失败，模型被迫继续尝试

**情况 C：模型跳过工具直接给答案（我们遇到的 DeepSeek 问题）**
- 这时不是返回 thought，而是返回 Final Answer
- 但这个 Final Answer 是**幻觉**——模型用训练数据或自身知识编造的
- 因为没调工具，答案不基于实际检索结果

**关键区分**：
- `parsed.thought` = 模型的思考过程（"我要做什么"）
- `parsed.final_answer` = 模型的最终回答（"答案是……"）
- 幻觉主要出现在 Final Answer，而不是 Thought

## 思考题 3：为什么 V4 要用 LangGraph 重写这个循环？

**答案**：V3 手写实现暴露了所有痛点，V4 用框架一一解决。

**V3（手写）的痛点**：

| 问题 | V3 怎么解决 | 痛点 |
|------|------------|------|
| 状态管理 | 手动拼 scratchpad 字符串 | 容易格式错误，不好扩展 |
| 工具调用 | 文本解析 + 正则提取 | 依赖模型"听话"，经常解析失败 |
| 错误处理 | 自己写 if/else | 容易遗漏边界情况 |
| 多模型切换 | 改环境变量 + 改代码 | 不灵活 |
| 可视化 | 自己写 trace 脚本 | 工作量大 |
| 持久化 | 手动写 JSONL | 无法恢复中断的 agent |

**V4（LangGraph）的解决方案**：

| 问题 | V4 怎么解决 |
|------|------------|
| 状态管理 | `StateGraph` 声明式，自动累积历史 |
| 工具调用 | `@tool` 装饰器 + 原生 function calling API |
| 错误处理 | 框架内置异常处理和重试机制 |
| 多模型切换 | `ChatOpenAI(model=...)` 一行换模型 |
| 可视化 | LangSmith 自动生成 trace 网页 |
| 持久化 | 内置 checkpoint，可恢复中断的 agent |
| 可组合 | 加新工具只需加一个 `@tool` 函数 |

**最核心的差异**：
- V3：模型输出自由文本 → 你解析 → 你执行
- V4：模型通过 API 返回结构化 `tool_calls` → 框架自动执行

## 思考题 4：除了 stop 序列，还有什么办法防止模型编造假 Observation？

**答案**：有 5 种方案，各有优劣。

| 方案 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **1. stop 序列**（已实现） | API 层截断，不让模型生成 "Observation:" 之后的内容 | 零成本，最干净 | 依赖 API 支持 stop 参数 |
| **2. 输出校验 + 重试** | 检测 raw_output 是否包含编造的 Observation，有就重新调用 | 不依赖 API 特性，通用性强 | 浪费 token 和时间（重试 = 再花一次钱） |
| **3. 输出截断/清洗** | 检测到 "Observation:" 就丢弃后面的内容 | 不浪费 API 调用 | 如果模型把 Observation 写在 Action Input 之前就麻烦了 |
| **4. 强化 prompt** | 加规则："你只能输出到 Action Input，禁止生成 Observation" | 简单 | 模型不一定听话 |
| **5. function calling API** | 模型返回结构化 tool_calls，不会编造结果 | 100% 可靠，协议层保证 | 需要 API 支持，V4 才用 |

**最佳实践**：stop 序列 + 输出截断兜底（双重保险 / 纵深防御）
- 第一层（API 层）：`stop=["Observation:"]` 让模型在生成时截断
- 第二层（代码层）：如果 API 没遵守 stop，代码再检测并截断

## 核心概念：stop 序列

### stop 是给谁用的？

**stop 不是给模型"看"的，而是给 API 服务端用的**。

```
你调用 API 时：
┌─────────────────────────────────────────┐
│  client.chat.completions.create(        │
│    messages=[...],  ← 这是给模型的输入  │
│    stop=["Observation:"], ← 这是给 API 的指令 │
│  )                                      │
└─────────────────────────────────────────┘
```

### stop 序列的工作流程

```
你的 Python 代码          API 服务端              模型
     │                      │                    │
     │── 发送请求 ──────────▶│                    │
     │   (带 stop 参数)      │                    │
     │                      │── 开始生成 token ──▶│
     │                      │◀── Thought: xxx ────│
     │                      │   (检查：不是 stop) │
     │                      │◀── Action: xxx ─────│
     │                      │   (检查：不是 stop) │
     │                      │◀── Action Input: xx │
     │                      │   (检查：不是 stop) │
     │                      │◀── Observati... ────│
     │                      │   ⚠️ 检测到 stop！  │
     │                      │   停止生成，截断返回 │
     │◀── 返回截断内容 ──────│                    │
     │                      │                    │
```

### 关键区分：各参数谁"看到"

| 参数 | 谁"看到" | 作用 |
|------|---------|------|
| `messages` | 模型 | 输入内容，模型据此生成回答 |
| `temperature` | 采样器 | 控制随机性，影响 token 选择概率 |
| `max_tokens` | API 服务端 | 限制输出长度，超过就截断 |
| `stop` | API 服务端 | 遇到特定字符串就截断 |

### 类比

- **模型** = 写作文的学生
- **messages** = 作文题目和要求（学生看到的）
- **stop** = 老师手里的红笔，看到某个词就画个句号让学生停笔（学生不知道）

### 为什么 stop 高效

如果是"生成完再检测"（你的代码检测）：
> 学生写作文 → 写完整篇 → 老师看到某个词就撕掉后面几页 → **浪费**

实际的 stop 机制（API 服务端检测）：
> 学生写作文 → 老师盯着学生写 → 写到某个词时老师立刻喊"停笔" → **后面根本不会写**

## 学习总结

### ReAct 的本质
- **Re**asoning（思考）：模型分析当前情况，决定下一步
- **Act**ing（行动）：执行工具获取信息
- **循环**：Thought → Action → Observation → Thought → ...

### 为什么 V3 有价值
1. **理解原理**：亲手实现"轮子怎么转"
2. **暴露痛点**：文本解析不可靠，所以知道为什么需要 function calling
3. **对比学习**：V4 不是凭空出现，而是解决 V3 的所有问题

### 学习路径
```
V3（手写 ReAct）
  ↓ 理解原理 + 暴露痛点
S2（function calling）
  ↓ 学会结构化工具调用
V4（LangGraph）
  ↓ 框架封装 + 工程化
生产级 agent
```
