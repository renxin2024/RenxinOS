## RAG 评测入门——从本项目学到的

### 大背景：为什么要评测？

RenxinOS 是一个 **RAG Agent**——从笔记中检索信息来回答问题。怎么知道它「搜得准不准」？

> **核心认知**：没有评测就没有优化方向。就像考试不是为了为难你，而是告诉你哪里不会。

**评测驱动的开发循环**：

```
设计评测题 → 跑基线 → 找瓶颈 → 针对性优化 → 再跑评测验证
    ↑                                              |
    └──────────── 持续迭代 ─────────────────────────┘
```

---

### 第一课：评测题类型学

评测题就像「考试题」——每道题测试检索系统的一种能力：

| 题型 | 测试什么 | 类比 |
|---|---|---|
| **单跳（single_hop）** | 关键词直接能搜到 | 开卷考试直接翻目录 |
| **多跳（multi_hop）** | 需要跨多个文件组合 | 需要翻好几章才能答 |
| **模糊（fuzzy）** | 用户用口语问，笔记用术语写 | 老师换了个说法出题 |
| **负例（unanswerable）** | 知识库里根本没有答案 | 超纲题——好学生会说「没学过」 |
| **同义改写（paraphrase）** | 同一个问题换种说法 | 同一道题用中文/英文/大白话问 |

**关键认知**：题不是越多越好，而是要**覆盖各种情况**。就像你不会只出 10 道加法题来考数学。

#### 出题技巧

- **单跳**：从某个 chunk 的核心内容反向构造问题。chunk 讲 4D → 问「什么是 4D 工作法？」
- **多跳**：找跨 2+ 文件的共同主题。任务打勾规则散布在多处 → 问「打勾写在哪里？」
- **模糊**：用口语化/间接表述替代笔记原文术语。笔记写「探索型无边界」→ 问「忍不住优化工具怎么办？」
- **负例**：故意问知识库中不存在的问题。如「如何用 LangChain 搭建 Agent？」
- **改写**：同一问题写 2-3 种表述，从含术语到完全口语化

### 第二课：基线思维

**做了什么**：用当前 keyword 检索跑全部题，得到分数

**为什么**：在任何优化之前，你需要一个**基线（baseline）**——现在的水平是多少？

> **核心概念**：基线就是「不改任何东西时的成绩」。将来换了 Embedding 模型，跟基线比就知道进步了多少。

**本项目的基线数据**：

| 版本 | 题数 | Recall@8 | KW Score | 说明 |
|---|---|---|---|---|
| V1（10 题旧版） | 10 | 88.3% | 91.5% | 只有 happy path |
| V2（16 题诊断版） | 16 | 61.5% | 71.3% | 加了负例+改写，暴露真实问题 |

> **反直觉**：分数降了反而是进步——因为新评测发现了旧评测没暴露的问题。

### 第三课：RAG 两阶段分离评测

RAG 系统分两步：**检索**（找资料）→ **生成**（写答案）。

```
用户提问 → [检索模块] → top-k 文档片段 → [LLM 生成] → 最终回答
              ↑                              ↑
          评检索质量                     评生成质量
       Recall@k / NDCG            Faithfulness / Correctness
```

**为什么分开评**：如果最终答案不好，你得分清是：
- 「没搜到正确文档」→ 改检索策略
- 「搜到了但 LLM 答错了」→ 改 prompt 或模型

**我们目前做了什么**：
- ✅ 检索侧评测（Recall@k、KW Score）
- ✅ 为每题写了 ground_truth_answer（标准答案）
- ⏳ 生成侧评测（等 V2 S5+ 引入 Faithfulness）

### 第四课：Pipeline 诊断——不要只看总分

参考了 **IBM 漏斗评测法**（arXiv:2410.12812）：

```
有效提问？ → 文档存在？ → 检索命中？ → 上下文够？ → 答案好？
    ↓             ↓            ↓            ↓           ↓
  过滤无效    补内容gap    修检索策略    改chunk策略   改prompt
```

**IBM 核心思路**：不看总分，看**每步 drop-off**。每次只聚焦漏斗中最大瓶颈。

**我们的评测脚本识别 3 种失败模式**：

| 失败模式 | 含义 | 本项目占比 | 修复方向 |
|---|---|---|---|
| `NONE` | 正常 | 69% | 无需修复 |
| `SEARCH_FAILURE` | 有正确文档但搜不到 | 19% | 改检索策略（Embedding） |
| `SHOULD_REFUSE` | 没答案但硬返回了 | 12% | 加 score 阈值拒答 |

> **调试思路**：先修占比最大的失败模式，再跑评测看变化。

### 第五课：鲁棒性测试——同义改写

**做了什么**：对关键问题写 2-3 种说法，看结果是否一致

**为什么**：企业级 RAG 需要处理用户的各种表述方式。这是**鲁棒性测试（Robustness Test）**。

**本项目结果**：

| 改写组 | 含术语版 | 口语化版 | 一致性 |
|---|---|---|---|
| 4D | ✅ 100%「怎么判断该做/委托/推迟/删」 | ❌ 0%「手头事太多怎么排」 | INCONSISTENT |
| decision | ✅ 100%「要不要写决策文档」 | ❌ 0%「碰到选择要不要单独想想」 | INCONSISTENT |

**核心洞察**：keyword 检索只认字面匹配。用户不说术语就搜不到。

→ 这正是 **Embedding（语义向量检索）** 要解决的问题——把「手头事太多」和「4D 优先级排序」映射到相近的向量空间。

---

### 第六课：负例题与幻觉防护

**做了什么**：加了 2 道知识库里不存在答案的题

**为什么**：好的 Agent 不仅要会回答，还要会说「我不知道」。硬编答案叫**幻觉（hallucination）**——企业级 RAG 的头号敌人。

**本项目结果**：
- Q11「如何用 LangChain 搭建 Agent？」→ 返回了 score=3.0 的无关结果
- Q12「Python 爬虫」→ 返回了 score=3.0 的无关结果

**修复方案**：设一个 score 阈值（如 < 5.0），低于时返回「知识库中未找到相关内容」

---

### 第七课：领域专属词 vs 通用词

**KW Score 陷阱**：如果关键词包含「任务」「方法」等通用词，很多无关文档也会命中 → 假阳性

**做法**：只用**领域专属词**——只有目标文件才会出现的词

| 差 | 好 |
|---|---|
| 「任务」「方法」「系统」 | 「铁律」「全局任务池」「30 秒清单」 |
| 「目标」「计划」 | 「Specific」「Measurable」「Time-bound」 |

### 知识图谱：RAG 评测学习路径

```
设计评测题（出题方法论）
    ↓
跑基线（基线思维）
    ↓
评测优化
├── ground_truth_answer → 解锁生成评测
├── 负例题 → 幻觉防护
├── pipeline 诊断 → 失败模式定位
├── 同义改写 → 鲁棒性测试
└── 领域专属词 → 消除假阳性
    ↓
针对性优化（Embedding / 阈值 / prompt）
    ↓
再跑评测验证 → 数据驱动决策
```

### 思考题（自测）

1. 为什么分数从 88.3% 降到 61.5% 反而是进步？
2. 如果你要加第 3 组同义改写，你会选哪个问题？为什么？
3. Embedding 接入后，你预测哪类题提升最大？哪类题可能不变？
4. 如果只改一个失败模式，你会先改哪个？为什么？
5. 「系统不会拒答」这个问题，除了加 score 阈值，还有什么别的方案？
